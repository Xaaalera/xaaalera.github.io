00Glossary — so the rest isn't a slog

TermIn plain words
SkillAn instruction file for Claude: "when doing X, work like this." Each has a description — a couple of sentences the model uses to decide whether to open it.
MarketplaceA git repository of skills that projects pull them from.
TriggeringDid the skill fire when needed, and stay silent when not.
Eval / probeAn automated triggering test: a fresh Claude instance gets one question — we watch whether it reaches for the skill. One probe = one such run.
Queryset18 questions per skill: 9 "should fire" + 9 "should stay silent."
Recall / precisionRecall — doesn't miss what it should catch; precision — doesn't butt in where it shouldn't.
Held-out testA slice of questions hidden from the "improver," used as the only judge — protection against teaching to the test.
HookA script the environment runs by itself on events (before a tool call, after a file write) — enforcement instead of self-discipline.

01How it started: "where are my tokens going?"

There was no plan to "build an eval system." There was an annoying fact: tokens were melting faster than seemed reasonable. I went digging, and the audit uncovered a layer cake:

  • oneSkills loaded twice. My personal marketplace and the work one are near-mirrors, and the project had both enabled: ~23 skills sat in context as two copies. Every session. Silently. The fix was one line of config; the savings — on every run since.
  • twoCLAUDE.md had bloated. Instruction files that always load: the global one, the project one (~1,500 tokens), plus communication-style rules duplicated in two places at once — double payment every session and version-drift risk.
  • three40+ skills had piled up — and not one had ever been through an eval. I had built a big skill base, but there was simply nothing to check descriptions with: no tool, no method, not a single measurement.

Then, the grown-up move: I ran my whole setup against Anthropic's official practices (their reference plugin repository) and got a gap list — missing evals being the fattest gap of all. That's how the consolidation plan was born: kill the double load, slim down the always-loaded files, add reviewers — and learn to measure skills with numbers. This page is about that last, bumpiest item.

02First a bicycle, then — "oh, wow"

Being an honest engineer, I naturally started building my own scorer — building it, of course, not with my own hands but with a Claude agent's. Remember that division of roles: it will matter later, when we hand out the blame. The scorer even worked — and gave every skill exactly 5.0/10. Every one. Identically. (Remember this trick: numbers that are too even are a signal.) And here, instead of fixing the bicycle further, I stopped and asked the question worth asking before the first line of code: doesn't Anthropic already have this?

Oh, wow — they did. A whole pipeline inside their skill-creator plugin that I hadn't even suspected:

  • run_eval.py — runs the questions and detects firings;
  • run_loop.py — an automatic improvement loop: measure → an LLM rewrites the description based on the misses → re-measure → pick the best version by the held-out test;
  • a methodology: questions must be realistic (paths, typos, laziness), and negatives must be "near-misses" only.
The decision: don't rewrite their pipeline and don't finish the bicycle — hybridize. Take Anthropic's loop wholesale and swap exactly one function: their code registers the skill under test as a slash command; I register it as a real skill — the way it lives in production. One substitution, zero forks, upstream updates for free.

03Side explosion: 240 agents on Opus

In parallel, an incident happened that paid for itself many times over. For a code review I launched a multi-agent workflow: "hunters find bugs, skeptics verify every finding." Half an hour later the counter read 240 agents — all on Opus. Not the most expensive model (that crown belongs to Fable), but a pricey one — and here two hundred and forty mouths were burning it at once.

  • →Each round, the hunters rephrased the same findings — title-based deduplication didn't catch them, and every "new" finding got two more verifiers. A snowball.
  • →The stop condition "until findings dry up" never arrives with a creative model — it always invents something.
  • →Nobody explicitly set a model for the agents — they all silently inherited the expensive session one.

Out of the incident came a three-layer system:

LayerWhat it does
The meta:model-routing skillA law of tiers: the expensive model only for a few "head" judges and synthesis, the mid-tier for judgment work (review, verification, code reading), the cheap tier for mechanics (greps, renames). Plus a rule: before any mass launch — measure one unit and present a scale table in quota percentages, not money: "this eats a quarter of the window" beats any price tag.
The guard hookA check before every agent launch: no explicit model — BLOCKED with an explanation. Discipline is a consumable; a hook is capital: within its first hour the guard blocked a call from its own creator.
Agent defaultsPermanent agents get their model pinned in config — every launch is cheap by construction. Along the way it turned out the "docs hunter" sat on a model too weak for its job — bumped it.

04Hunting the measuring stick: five bugs

Then, a detective story. Every bug masqueraded as "the skills are bad" while the measuring stick was the culprit. The catching technique was the same every time: suspiciously even numbers → hand-check one case.

#1
The parser read ">" instead of the description
Symptom
Every skill — exactly 5.0/10. The "before → after" line never prints.
False lead
"The descriptions are weak, rewrite them."
Cause
YAML supports "folded" multi-line descriptions (description: > with text below). The agent's parser grabbed only the > character — an entire day was spent measuring empty descriptions. The spicy part: Anthropic's parser handled this from the start; the agent introduced the bug itself while "simplifying" their code.
Fix
Unfold block scalars: 
# before: capture the rest of the line after "description:"
m2 = re.search(r"^description:\s*(.*)$", frontmatter, re.M)
return m2.group(1).strip()          # for a block this is literally ">"
# after: unfold the block scalar from the lines below
if inline in (">", "|", ">-", "|-"):
    body = [ln.strip() for ln in continuation_lines(frontmatter)]
    return " ".join(body)           # the full description text
Check
The old (from git) and new descriptions both parse in full and differ.
#2
The detector judged by the first move
Symptom
17 skills out of 23 — zero recall. Probes finish suspiciously fast (~10 sec).
False lead
"Rule-type skills just don't fire in a vacuum."
Cause
"First tool decides" logic (also inherited from upstream): if the model's first move was drafting a plan or opening files, the probe was instantly recorded as "didn't fire" — even though the skill was invoked on the second move. Knowledge questions ("how does X work") survived — the skill comes first there; action requests ("do Y") all died.
Fix
Watch the probe's whole conversation to the final event; a tripwire comment describing the incident is baked into the code: 
# before: the first "foreign" tool is a verdict
if cb.get("name") in ("Skill", "Read"):
    pending_tool = True
else:
    return False        # a plan/ls as the first move kills the probe
# after: track every tool_use until the result event
is_probe_tool = cb.get("name") in ("Skill", "Read")
pending_tool, accumulated = is_probe_tool, ""
# Do NOT restore upstream's first-tool-decides shortcut:
# ...misscored 17 skills as 0-recall (2026-07-04 incident)
Check
A unit test on a synthetic stream: plan as the first move, skill as the second → "fired."
#3
A twin was sitting in the sandbox
Symptom
Detector fixed, old skills still at zero — while two of them somehow score a healthy 92–94%.
False lead
"So the descriptions really are weak."
Cause
Probes run in an "empty" temp folder, but globally-enabled plugins load everywhere. The model saw two identical skills — the real one from the marketplace and the probe copy — and invoked the real one. The probe "missed." The pearl of evidence: the two healthy skills were exactly the ones with no global mirror. Note the echo: this is the same double-loading that started the whole audit — it came back to poison the measurements too.
Fix
The sandbox writes its own .claude/settings.json, disabling every plugin from every marketplace in the local cache. The final isolation philosophy: "production as is, minus the twins" — hooks, global context and MCP servers stay alive, because in reality a description fights for the model's attention in exactly that noise. 
Path(sandbox, ".claude", "settings.json").write_text(json.dumps({
    "enabledPlugins": {p: False for p in _all_known_plugins()},
}))
# _all_known_plugins() enumerates ~/.claude/plugins/cache/*/*
# — source-independent: a plugin enabled at ANY level
#   is dead inside the sandbox
Check
A zero-recall skill came back to life: train 12/12, test 5/6.
#4
Editing under the wheels
Symptom
A fleet re-run — zeros for everyone again, even yesterday's healthy ones. 23 skills "checked" in 12 minutes.
False lead
"The quota choked it."
Cause
Right in the middle of the run, the agent was hardening the isolation and renamed a function without updating the call site. Every probe died with a NameError and was silently recorded as "didn't fire." A half-finished edit is worse than no edit. 
def _all_known_plugins():          # renamed...
{p: False for p in _user_enabled_plugins()}  # ...call site left behind
                                   # NameError in every probe
Fix
One line. The rule, forever: you don't edit a running pipeline; an interrupted edit gets reverted or finished to a passing unit test before anything runs on top of it.
Check
The control skill returned to its numbers: train 24/24.
#5
A probe failure masqueraded as "didn't fire"
Symptom
Third run: the first 12 skills in the queue — lively, varied numbers; everything from #13 on — flat zeros. The log shows it: before the boundary a skill took ~2 minutes, after — 10 seconds each.
False lead
"Skills from one domain are worse than the rest" (the zeros landed exactly on one group).
Cause
After ~25 minutes of non-stop probing, the API started rejecting requests with a burst rate limit. And in the measuring stick, a probe failure was indistinguishable from an honest "the skill didn't fire" — rejected probes silently fell into the zeros.
Fix
A probe-error class: API failure → retry with pauses; after the third failure — a loud flag, but never a silent zero. Plus pauses between skills in the queue. 
if event.get("is_error") or subtype != "success":
    raise ProbeError(...)   # an API failure is NOT "didn't fire"

for attempt, pause in enumerate((0, 20, 60)):   # retries
    ...
print("UNRELIABLE RESULT: N probe(s) errored after retries")
Check
Unit test: an error result from the API raises an exception instead of returning False. The 11 killed skills were queued for a re-shoot after the quota reset.
The moral of all five: not one bug was found by "reading the code." All were found the same way — the numbers looked too even, and I went to hand-check a single case. Distrusting your own results isn't paranoia; it's part of the instrument. And the second moral: four bugs out of five lived where the agent had "simplified" or half-finished something; Anthropic's code, taken as is, never failed once.

05What an honest stick showed

The first full cycle — on the newborn meta:model-routing skill (the one born from the 240-agent incident):

56%
recall of the original description
94%
after one improver iteration
0
false fires before and after

Anthropic's improver did the key thing right: it didn't memorize the failed questions — it generalized the intent, writing the situations themselves into the description, and the skill started catching phrasings that were never in the training set:

# before
Use whenever spawning subagents or authoring a Workflow script…
# after (excerpt)
…ANY multi-agent work, not just when models are mentioned.
Triggers: "spawn/kick off subagents", "a bunch of agents in parallel",
fleets that audit/comb/sweep/hunt across files, loop-until-done runs…

And the first honest fleet triage (the half that got measured before bug #5) immediately painted a working picture: a healthy five with 83–94% recall at zero false fires, and a treatment queue of skills whose descriptions "retell the contents" instead of saying "when to call me." Exactly the knowledge the whole thing was started for: now I know what to fix and in what order.

06Anthropic best practices I adopted

PracticeThe gist
Train/test split60% of the questions go to the improver, 40% get hidden. Pick the winner only by the hidden ones — otherwise you get a description tuned to the exam questions.
Negatives = near-missesTest a skill's silence on questions that share its keywords but belong to a neighbor. "Write a Fibonacci function" as a negative for a PDF skill tests nothing.
Realistic questionsWith file paths, component names, laziness and typos — the way real people type, not the way specs are worded.
Description = "when," not "what"The model reads the description to decide whether to open the skill. Retelling the contents is wasted space; you need firing conditions, and pushy ones.
Generalize, don't enumerateWhen improving a description from its failures, turning it into a list of specific questions is forbidden — that's bloat and overfitting.
Rationalization tablesIn rule-type skills: a column "the thought you'll excuse yourself with" → a column "reality." Catches corner-cutting better than ten ALL-CAPS "NEVER"s.
Progressive disclosureA short skill + depth in a references/ subfolder loaded on demand — long walls of text don't sit in context for nothing.

07The protocol going forward

Money and scale

  • Measure one, then multiply. Before any mass launch, run a single unit with a real token counter. Gut estimates are banned: one of them missed by 3× that very day.
  • Percentages, not money. Scale is presented as a table: tokens by type × count × share of the 5-hour quota window. "Eats a quarter of the window" beats any price tag.
  • A new skill gets the full cycle immediately (born calibrated, a one-off cost). Mass-checking old ones — a cheap triage for all, the treatment loop only for the failures.

Measurement hygiene

  • Validate your measuring stick on a case representative of the real workload: mine passed its check on a new skill — and silently lied about the old ones, which had twins.
  • Inventory the environment: a single debug print of "what does the model actually see in the sandbox" would have found the twins within the first hour.
  • A measurement error ≠ a negative result. A probe failure must scream, not quietly land in the zeros — otherwise a rate limit looks like "half the skills are broken."
  • Suspiciously even numbers are a signal to hand-check one case before building theories.
  • You don't edit a running pipeline.

Guards instead of discipline

Over the day the agent's discipline slipped more than once: forgot to set a model for subagents, left an edit half-finished. Every such case is now closed by a hook that checks by itself and blocks with an explanation — and the new guards caught the very agent that wrote them within their first hour. Which means they stand exactly where they should.
